# 项目学习资料

## 项目架构解析

系统分为后端（Spring Boot）、后台管理（Vue2 + Element）、移动端（Vue2 Web）三块，前后端通过 RESTful JSON 接口通信，认证用 JWT。可以类比成「一家公司」：后端是总部机房负责数据和规则，后台是办公室电脑上的管理界面，移动端是手机上的办事入口，三者共用同一套接口和数据库。

## 登录流程

用户输入账号密码 → 前端调用 /auth/login → 后端校验用户名密码（密码为 MD5 比对）→ 通过则生成 JWT 并返回 token、username、realName、roleCode → 前端把 token 和用户信息存入 Vuex 与 localStorage → 之后每次请求在请求头带上 Authorization，后端拦截器解析 token 并注入当前用户到线程上下文，完成「登录态」维持。生活例子：进门时在前台登记拿门禁卡（token），之后进每个房间都刷卡，门禁系统（拦截器）识别你的身份。

`localStorage` 就是浏览器给每个网站提供的**“本地永久小仓库”**，

**为什么登录要存两份？** 因为如果你只存 Vuex，用户登录后一按 `F5` 刷新，Vuex 瞬间清空，用户就直接被踢回登录页了（这体验会让人抓狂）。所以我们要把 Token 备份在 `localStorage` 这个“硬盘”里。

Authorization **为什么要这么写**

- **端负责“带 token”**（写到 `Authorization` 里）。
- **后端负责“验 token”**（从 `Authorization` 读出来，验证是否有效/过期）。
- 你遇到的 **401**，通常就是这里（后端拦截器）判定：**没带、带错、过期、无效**。

## Token 原理

JWT 是一段签名的 JSON，内含用户标识和过期时间。服务端用密钥签名，客户端只持有不解析；请求时把 token 放在 Header，服务端用同一密钥校验签名并解析出用户信息，无需在服务端存 session。就像一张带防伪的「临时通行证」，谁持有谁就被视为对应用户，过期即失效。

## 权限控制原理

本项目采用简化 RBAC：用户有 role_code（如 ADMIN、USER），角色在 sys_role 表中有 permissions 字符串。后端在需要权限的接口里从 UserContext 取当前用户角色做判断（如角色列表仅 ADMIN 可访问）；前端可根据 roleCode 控制菜单或按钮显示。真正安全依赖后端校验，前端只是体验层「藏起来」无权限功能。

## 前后端通信流程

前端用 Axios 发 HTTP 请求，baseURL 指向 /api，开发时由 vue.config.js 的 proxy 转发到后端 8082。请求拦截器自动加 Authorization；响应拦截器统一处理 200 与 401（未登录则清空本地并跳转登录页）。生活例子：所有「快递」都走同一辆配送车（Axios），车上统一贴单号（Token），收到退件（401）就统一退回发货地（登录页）。



*我们这里 token 存在 Vuex 里是为了让页面和 Axios 拦截器能随时读取；同时把 token 写入 localStorage 里是为了刷新后仍然保持登录态。Cookie 是另一种持久化方案，但本项目没有用它，主要是考虑实现简单与安全策略；如果用 cookie 需要额外处理 CSRF 等安全问题。*

## 项目亮点

- 前后端分离、JWT 无状态认证、Swagger 文档齐全，便于面试演示和联调。  
- 后台首页 ECharts 展示业务数据分布，体现数据可视化能力。  
- 同一套接口服务后台与 App，体现「多端复用」思路。  
- 无外键、有初始数据、命名与规范统一，适合零基础学习和简历展示。

## 如何讲项目

建议结构：先一句话说明系统做什么（个人效率管理：待办+日程）；再说技术栈与分工（Spring Boot + Vue2 双端）；然后挑 2～3 个点深入：例如登录与 JWT、路由守卫与权限、ECharts 与统计接口；最后说遇到的问题与解决（如 Swagger 路径匹配、日期格式统一）。控制在 2～3 分钟，突出「你做了什么、用了什么技术、解决了什么问题」。
