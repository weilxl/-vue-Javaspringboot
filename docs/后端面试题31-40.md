# 后端面试题 31-40

## 31. 什么是「依赖注入」？
对象所需的依赖（如 Service、Mapper）不由自己 new，而是由容器（Spring）创建并「注入」进来。这样解耦、易测试、易替换实现。本项目 Controller 依赖 Service、Service 依赖 Mapper，都在构造器里声明，Spring 自动注入。就像公司配电脑：你不用自己买，行政（容器）给你配好。

## 32. 如何理解「面向接口编程」？
Service 层定义接口，实现类加 @Service，Controller 里依赖接口类型。这样可随时换实现（如换缓存、换算法）而不改调用方。本项目 TodoTaskService 是接口，TodoTaskServiceImpl 是实现，Controller 里注的是 TodoTaskService，符合面向接口编程。

## 33. 什么是「Bean」？怎么注册？
Bean 是交给 Spring 管理的对象，通常带 @Component、@Service、@Controller 等注解，或由 @Bean 方法返回。Spring 启动时扫描并创建这些对象，放到容器里，需要注入时就从容器取。本项目 Controller、Service、Mapper 接口对应的实现都是 Bean，由扫描或 MapperScan 注册。

## 34. 配置文件 application.yml 和 application.properties 的区别？
都是配置，yml 层级清晰、可读性好，properties 是 key=value。本项目用 yml 配置端口、数据源、JWT、mybatis-plus、pathmatch 等，一处修改即可，无需改代码。

## 35. 如何理解「约定优于配置」？
Spring Boot 很多默认行为：如默认扫描主类所在包、默认 8080 端口、默认找 application.yml。只要按约定放好文件、写好注解，很少要手写配置。本项目结构符合约定，所以只写了必要配置，其余用默认即可。

## 36. 什么是「线程安全」？在 Spring 里要注意什么？
多线程同时访问同一变量若不加保护可能出错。Spring 里 Controller、Service 默认是单例，不要用可写的成员变量存请求相关数据；当前用户应放 ThreadLocal（如 UserContext），每请求一线程，互不干扰。本项目 UserContext 即 ThreadLocal，不会串请求。

## 37. 如何做「参数校验」？
用 JSR-303/380：在 DTO 上 @NotBlank、@NotNull、@Size 等，方法参数加 @Valid，校验失败抛 MethodArgumentNotValidException，由全局异常处理转成统一错误。本项目 LoginRequest、RegisterRequest 已用，其它入参也可按需加注解。

## 38. 什么是「跨域」？后端如何解决？
浏览器限制：前端页面的域名/端口与请求的接口不一致时，默认会拦截响应。后端解决：在响应里加 CORS 头（Access-Control-Allow-Origin 等），允许指定来源。本项目 WebConfig 里 addCorsMappings 配置了允许的来源和方法，开发时前端可直接调后端。

## 39. 如何理解「分层架构」的好处？
每层只关心自己的事：Controller 管协议、Service 管业务、Mapper 管数据。改接口不影响业务，改表结构主要动 Mapper，易维护、易测试、易协作。就像公司分部门：前台、业务、仓库各干各的，流程清晰。

## 40. 为什么要用 DTO 而不是直接用 Entity 接收参数？
Entity 和表结构强绑定，字段多、有 id/createTime 等，直接暴露给前端容易误改或泄露；DTO 只包含本次接口需要的字段，清晰、安全。本项目登录用 LoginRequest、RegisterRequest，没有用 SysUser 直接接参，符合常见做法。若以后做「修改用户」可再定义 UpdateUserDTO。
