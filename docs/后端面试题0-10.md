# 后端面试题 0-10

## 1. Spring Boot 和传统 Spring 相比有什么好处？
传统 Spring 要配大量 XML 或 Java 配置，整合 Tomcat、数据源等较繁琐。Spring Boot 提供起步依赖（starter）和自动配置，内嵌 Tomcat，写一个 @SpringBootApplication 的 main 就能跑起来，大大减少配置量。本项目直接运行 EfficiencyApplication 即可启动完整 Web 服务，就像「一键启动」的集成环境。

## 2. 什么是 Spring MVC？请求怎么到 Controller？
Spring MVC 是 Web 层的框架：请求先到 DispatcherServlet，根据 URL 和 @RequestMapping 映射到对应 Controller 方法，方法里调 Service、返回结果，最后由消息转换器转成 JSON。本项目 AuthController、TodoTaskController 等就是 Controller，负责接收参数、调用 Service、返回 ApiResponse 包装的结果。

## 3. MyBatis 和 MyBatis Plus 的区别？
MyBatis 需要手写 SQL 或注解，做 CRUD 也要写。MyBatis Plus 在 MyBatis 基础上封装了 BaseMapper，单表 CRUD 不用写 SQL，用 LambdaQueryWrapper 等写条件即可。本项目用 Plus，Mapper 继承 BaseMapper，Service 里用 selectList、insert、updateById、deleteById 等，只有统计在 Service 里用 stream 聚合，无需写 XML。

## 4. JWT 是什么？和 Session 有什么区别？
JWT 是一段签名的 JSON 字符串，内含用户标识、角色、过期时间等，服务端用密钥签名，客户端携带请求，服务端验签即可识别用户，无需在服务端存 session。Session 是服务端存一份，客户端只带 sessionId。JWT 无状态、易扩展、适合前后端分离；Session 便于「立刻失效」。本项目用 JWT，登录后签发，后续请求 Header 带 token，拦截器验签并注入用户信息。

## 5. 如何实现「登录拦截」？
用 HandlerInterceptor：在 preHandle 里从请求头取 token，校验是否有效、是否过期，通过则把用户信息放入 ThreadLocal（如 UserContext），返回 true 放行；不通过则写 401 响应并返回 false。本项目 JwtAuthInterceptor 即如此，对 /auth/login、/auth/register 和 Swagger 路径放行，其余路径都要带合法 token。

## 6. 为什么用 ThreadLocal 存当前用户？
每个请求一个线程，ThreadLocal 是线程隔离的，请求进来时 set 当前用户，Controller 里 get 就能拿到，请求结束 afterCompletion 里 clear，避免线程复用时把「上一个请求的用户」带到下一个请求。生活例子：银行窗口，每个窗口（线程）只处理当前客户（用户），不会拿错别人的单子。

## 7. 统一返回格式有什么好处？
前端只需判断 code 和取 data，不用每套接口各写一套解析逻辑。本项目 ApiResponse 包装 code、message、data，成功 200、失败 500、未授权 401，前端拦截器统一处理，业务只关心 data。就像快递单：格式统一，收件人只看「收件人」和「物品」两栏即可。

## 8. 参数校验 @Valid 怎么用？
在 DTO 上对字段加 @NotBlank、@NotNull 等注解，Controller 方法参数前加 @Validated 或 @Valid，请求进来时自动校验，不通过会抛 MethodArgumentNotValidException，由全局异常处理器转成统一错误返回。本项目 LoginRequest、RegisterRequest 用了 @NotBlank，AuthController 用 @Validated，校验失败会返回 500 和具体字段错误信息。

## 9. 全局异常处理怎么做？
用 @RestControllerAdvice + @ExceptionHandler：针对 MethodArgumentNotValidException、RuntimeException、Exception 等分别处理，返回 ApiResponse.fail(message)。这样 Controller 里不用 try-catch，业务异常直接 throw new RuntimeException("xxx") 即可，由统一出口返回给前端。就像客服中心：所有投诉都转到一个部门，由他们统一回复格式。

## 10. 为什么数据库不用外键？
教学和简历项目更注重「好理解、好改」：无外键时表之间靠业务字段约定关联，改表结构、导历史数据更灵活，不会因外键约束卡住。真正安全靠业务代码和事务保证一致性。本项目 todo_task、schedule 有 owner_username 但不建外键，和 sys_user 的关联在业务层理解即可。就像两家公司合作：不签死合同（外键），靠约定（业务字段）协作，灵活性更大。
